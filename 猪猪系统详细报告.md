## 三、核心模块说明

### 3.1 对话服务 (`llm_service_minimax.py`)
```python
# 调用MiniMax API进行对话
chat_with_minimax_async(
    message=用户消息,
    model="MiniMax-M2.5-highspeed",
    history=对话历史,
    system_prompt=人格设定
)
```

### 3.2 记忆服务 (`memory_service.py`)

#### 向量检索流程:
```python
1. 用户消息 → get_embedding() → 向量(1024维)
2. 加载所有记忆 → 计算余弦相似度
3. 综合评分 = 向量分×0.4 + 重要度×0.2 + 类型权重×0.2 + 时间权重×0.1 + 命中权重×0.1
4. 排序返回Top5
5. 过滤掉问题形式(带"？")的记忆
6. 将有效记忆拼接到用户消息后面
```

#### 记忆类型:
| 类型 | 权重 | 说明 |
|------|------|------|
| user_info | 0.9 | 用户信息(名字等) |
| user_preference | 0.9 | 用户偏好(爱好等) |
| project | 0.8 | 项目信息 |
| goal | 0.7 | 目标计划 |
| reflection | 0.6 | 反思总结 |
| context | 0.5 | 工作上下文 |

#### 保存逻辑:
```python
# 自动判断是否保存
should_save(content):
    - 长度>5字符
    - 包含关键词(我叫/喜欢/项目/目标等)
    - 强制保存模式
```

### 3.3 API接口

| 接口 | 方法 | 功能 |
|------|------|------|
| `/api/chat` | POST | 普通聊天 |
| `/api/pig/send` | POST | 猪猪聊天(带记忆) |
| `/api/pig/memory/read` | GET | 读取记忆 |
| `/api/pig/memory/write` | POST | 写入记忆 |
| `/api/pig/trigger_reflection` | POST | 触发反思 |
| `/api/pig/personality` | GET/POST | 人格管理 |

## 五、记忆系统详解

### 5.1 存储结构
```json
{
  "id": "mem_1234567890",
  "content": "用户叫FK，最喜欢吃火锅",
  "type": "user_info",
  "importance": 0.9,
  "embedding": [向量数组...],
  "tags": [],
  "created_at": "1234567890",
  "last_hit": "1234567890",
  "hit_count": 1
}
```

### 5.2 检索流程图
```
用户发送消息
      │
      ▼
生成查询向量(1024维)
      │
      ▼
遍历所有记忆计算相似度
      │
      ▼
综合评分排序(0.25阈值)
      │
      ▼
过滤问题形式(带？)
      │
      ▼
拼接有效记忆到提示词
      │
      ▼
调用MiniMax API
```

### 5.3 调用示例
```python
# /api/pig/send 接口内部
async def pig_send_message(request):
    # 1. 检索相关记忆
    memories = await search_memories(user_message, top_k=8)
    
    # 2. 过滤有效记忆
    filtered = [m for m in memories if m["score"] > 0.25 and "？" not in m["content"]]
    
    # 3. 构建增强消息
    if filtered:
        context = "\n[记忆]\n" + "\n".join([m["content"] for m in filtered[:3]])
        full_message = user_message + context
    
    # 4. 保存新记忆
    await add_memory(user_message, force=True)
    
    # 5. 调用LLM
    response = await chat_with_minimax(full_message, ...)
```

我先给结论：

总评

7.8/10，已经能用，但还不是“长期稳定版”。

优点很明显：

架构简单，能跑，没过度设计

MiniMax 主脑 + Ollama 向量，这个分工对

记忆检索流程已经成型

但问题也很明显：

现在更像**“带记忆增强的聊天壳”**

还没到你最初想要的那种“长期成长型 AI 前端系统”



---

先说它做得好的地方

1）技术路线选得稳

后端 FastAPI

前端原生 HTML/CSS/JS

记忆 JSON + 本地向量

MiniMax 做主回复


这套东西的好处就是：

简单、透明、好修。

你现在最需要的不是炫技，而是：

能跑

容易查错

容易继续加功能


这点它做对了。


---

2）记忆系统已经不是假的

它不是只“保存聊天记录”，而是已经开始做：

分类

embedding

相似度检索

综合打分

过滤问题型记忆

记忆拼接进提示词


这说明它至少理解了：

记忆 ≠ 历史记录

这一点很关键。


---

3）接口设计已经预留了“小猪接管位”

这几个接口很有价值：

/api/pig/send

/api/pig/memory/read

/api/pig/memory/write

/api/pig/trigger_reflection

/api/pig/personality


这意味着后面你可以让真正的小猪：

直接读写这个前端系统

让它变成“小猪的外置人格壳/对话壳”


这个方向是对的。


---

现在最大的问题，我直接给你挑明

1）它现在的“长期记忆”还偏粗糙

现在逻辑是：

用户发言

检索记忆

拼接上下文

强制保存用户消息


这里最大的问题是这句：

await add_memory(user_message, force=True)

这很危险。

为什么危险

这会导致：

很多垃圾消息也进长期记忆

情绪句、废话句、一次性上下文都可能被永久存

时间一长，记忆库会脏得很快


我建议

应该改成三层：

即时记忆：先缓存

候选长期记忆：满足条件才进

反思确认后长期记忆：重要信息再正式固化


也就是说：

不是所有用户话都 force=True。


---

2）记忆拼接方式太原始

现在它是：

full_message = user_message + context

这能跑，但不够稳。

问题

这样很容易：

污染用户原始输入

让模型分不清什么是用户刚说的，什么是记忆

以后加多模态/搜索时会越来越乱


更好的方式

应该做成明确结构：

用户当前消息:
...

相关长期记忆:
1. ...
2. ...
3. ...

请基于当前消息优先回答，必要时参考长期记忆。

最好放进 system / memory context 结构里，而不是简单字符串拼接。


---

3）人格系统现在还是“写死 prompt”

你现在的 SYSTEM_PROMPT 还只是静态设定：

叫猪猪

活泼

友好

用 emoji

使用记忆


这不算“人格成长”，这只是固定人设。

你最初要的是

每日反思

成长增量

人格基线

冲突保护


现在报告里提到了未来方向，但还没落地。

所以这块现在不能算完成，只能算： “人格外壳已建立”


---

4）记忆存储用 JSON，v1 可以，长期不行

现在 JSON 文件没问题，v1 用这个很对。
但你要知道它的上限：

数据量大了会慢

并发写入容易出事

去重和更新会越来越麻烦

后面做反思、成长、多会话关联时会难受


结论

v1 保持 JSON 没问题。 但 v2 最好升级成：

SQLite / PostgreSQL 存结构化数据

向量单独存索引或专门库



---

5）测试 10/10 不能说明“长期稳定”

这个测试说明：

基础记忆闭环通了


但还不能证明：

长期记忆不会污染

多轮复杂任务不会串

并发下不会乱

记忆命中不会漂移

热门上下文不会覆盖旧重要记忆


所以这个测试是： MVP 通过，不是长期稳定通过。


---

我会建议你下一步怎么干

第一优先级：先修 4 个地方

你可以直接让小猪先改这 4 项，不要急着加新功能。

1. 取消“所有用户消息强制进入长期记忆”

改成：

先判断类型和重要度

普通聊天不直接入长期记忆

只入候选记忆池

重要信息才正式写入


2. 改进记忆注入格式

不要 user_message + context 这种直接拼接。
改成结构化 prompt 块。

3. 给记忆加“来源”和“状态”

建议每条记忆再加字段：

source: user / reflection / project / pig

status: temp / confirmed / archived


这以后会非常好用。

4. 增加“会话级短期记忆”和“长期记忆”分离

现在看起来它更像“长期记忆单层结构”。

你应该让它分：
